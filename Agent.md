# Agent.md — Job Hunt Dashboard (find_work)

Документ для ИИ-агентов и разработчиков: что строим, что уже есть, что делать дальше. Обновляйте статусы по мере прогресса.

---

## 1. Цель продукта

Закрытый **админский** веб-сервис (доступ **только с известного домена** + **API-ключ**): дашборд в духе hh.ru для поиска вакансий, сохранения в **PostgreSQL**, периодического обновления, аналитики, резюме, генерации сопроводительных писем и **анализа вакансии** через LLM (GigaChat / GPT и т.п.) с **ограничением частоты** (например, не чаще раз в 3 минуты на задачу/глобально — настраивается в UI).

Дополнительно: **фоновый ИИ-анализ «резюме ↔ вакансия»** по расписанию (например **раз в час**): выставляется **балл совместимости**, **категории/теги** (структурировано из LLM), **советы по вакансии** от лица **HR lead / IT-рекрутера** (тон и роль — в промпте). При балле **выше порога** (настраиваемо, ориентир **80–90**) — **уведомление в Telegram**. На фронте — отдельная **вкладка «Уведомления»**: накопление таких срабатываний с кратким объяснением и действием **«Прочитано»** (состояние хранится в БД).

## 2. Текущее состояние репозитория

| Компонент | Статус |

### 2.1 Фронтенд — что уже есть (`front/`)

**Стек:** Next.js 16 (App Router), React 19, TypeScript, Tailwind 4, shadcn/ui (Radix), `pnpm-lock.yaml`, без TanStack Query (пока локальный `useState` + моки).

**Маршруты и экраны:**

| Путь | Назначение |
|------|------------|
| `/` | Редирект/лендинг (при необходимости) |
| `/dashboard` | Обзор — данные с API (`/analytics`, `/vacancies`, `/searches`) |
| `/dashboard/vacancies` | Список с API, PATCH избранного, анализ и сопроводительное через LLM |
| `/dashboard/search` | CRUD сохранённых запросов через API |

**Контракт данных (ориентир для API):** `front/lib/types.ts` — `Vacancy`, `Resume`, `SearchQuery`, `Settings`, `Analytics`. Поля вакансии включают `externalId`, `source: 'hh'`, ЗП, график, `skills[]`, `status` (`new` \| `viewed` \| `applied` \| `rejected` \| `interview`), `isFavorite`, `isAnalyzed`, `matchScore`, `aiAnalysis`, `coverLetter`, опционально `responses` / `views` (в моках заполнены; из публичного HH могут отсутствовать — бэкенд отдаёт `null` / не включает).

**Авторизация UI:** `lib/auth-context.tsx` — проверка ключа через `GET /api/me`.

**Уведомления:** хедер — счётчик непрочитанных с `/api/notifications`; вкладка `/dashboard/notifications`; синк по кнопке обновления в хедере — `POST /api/sync`.

**Итог:** клиент ходит на **`/api/*`** (относительный путь; за nginx тот же origin). TanStack Query по желанию позже.

### 2.2 Бэкенд под этот фронт

- **Приоритет:** ответы API по смыслу и именам полей **совместимы с `front/lib/types.ts`** (при расхождении snake_case ↔ camelCase — Pydantic `Field(alias=...)` или единый JSON в camelCase).
- **Эндпоинты (маппинг на экраны):** вакансии CRUD/лист + PATCH статуса/избранного; поисковые запросы CRUD + триггер синка; резюме GET/PUT; настройки GET/PATCH (секреты LLM не отдавать обратно целиком — маскирование); аналитика агрегатами как `Analytics`; сопроводительное и анализ — POST по `vacancy_id`; синк — POST/статус; уведомления — по спеке выше.
- **Аутентификация:** тот же заголовок ключа, что задуман для продукта (`X-API-Key` / Bearer), валидируется только на сервере; фронт после логина подставляет ключ во все запросы.

## 3. Целевой стек

- **Backend:** Python 3.11+, FastAPI, SQLAlchemy 2 + **asyncpg** (или sync + psycopg), **Alembic — обязательно** (все изменения схемы только через ревизии; `alembic upgrade head` в Docker/CI), Pydantic v2, **`httpx`** для вызовов hh.ru **по образцу `main.py`**, `logging`, ruff/black, type hints.
- **Frontend:** Next.js (App Router), React, shadcn/ui, Tailwind — уже в `front/`.
- **БД:** PostgreSQL.
- **Прокси:** nginx (TLS, заголовки безопасности, проксирование `/api` → FastAPI, `/` → Next).
- **Планировщик:** APScheduler внутри FastAPI **или** отдельный worker + Redis (MVP — APScheduler в одном процессе с ограничениями).
- **Docker (обязательно к добавлению) — всё поднимается одной командой:**
  - **Цель:** `docker compose up -d` (или `docker compose up --build`) поднимает **полный стек**: **PostgreSQL**, **FastAPI**, **Next.js (прод-сборка)** и **nginx** как единую точку входа снаружи.
  - **`docker-compose.yml` — сервисы (минимум):**
    - **`db`** — официальный образ `postgres` (том для данных, пароль из env).
    - **`api`** — образ из `Dockerfile` бэкенда (uvicorn/gunicorn), сеть только внутри compose, порт наружу **не обязателен** (достаточно `api:8000` для nginx).
    - **`web`** — образ из `Dockerfile` каталога `front/`: **внутри образа выполняется прод-сборка фронта**, не `next dev`.
    - **`nginx`** — отдельный контейнер: конфиг **`server_name`** и при необходимости пути к сертификатам **подставляются из `.env`** (не хардкодить домен в репозитории). Типовой приём: шаблон `nginx.conf.template` + **`envsubst`** в entrypoint контейнера, переменные из compose (`env_file: .env`). TLS (или HTTP только в dev), `proxy_pass` на `web:3000` для UI и на `api:8000` для `/api`.
  - **Фронт в Docker — «как правильно» (прод):**
    - Multi-stage **`Dockerfile`** в `front/`: стадия **deps** (`pnpm install --frozen-lockfile` или `npm ci` — в репозитории есть `pnpm-lock.yaml`, зафиксировать один менеджер), стадия **build** (`pnpm run build` / `npm run build` с `ARG`/`ENV` для `NEXT_PUBLIC_*` при необходимости), стадия **run** — только артефакты + `node` и команда **`pnpm start` / `npm run start`** (это обёртка над `next start` для production).
    - **Не** использовать `next dev` в основном compose-файле продакшена — только для override под локальную разработку.
    - Опционально в `next.config`: `output: 'standalone'` — меньший финальный слой и явный `node server.js` вместо полного `node_modules` (уточнить по доке Next для вашей версии).
  - **`Dockerfile`** бэкенда: Python 3.11+, non-root user, кэш слоёв зависимостей.
  - **`.dockerignore`** для корня API и `front/`, чтобы не тащить `.venv`, `node_modules`, `.git` в контекст сборки.
  - **Переменные:** `.env.example` + проброс через `env_file` / secrets compose (`DATABASE_URL`, `ADMIN_API_KEYS`, LLM — не коммитить).
  - **Домен из `.env` (обязательная мелкая настройка):** одна переменная, например **`PUBLIC_DOMAIN`** или **`APP_HOST`** (значение вида `jobs.example.com` **без** схемы), плюс при необходимости **`PUBLIC_URL`** = `https://jobs.example.com` для ссылок в письмах/TG. Эти значения: **nginx** (`server_name`, `proxy_set_header Host`); **FastAPI** `CORS_ALLOWED_ORIGINS` / `PUBLIC_URL`; **сборка Next** — `NEXT_PUBLIC_APP_URL` или аналог (build-arg из того же `.env`), чтобы фронт знал публичный origin. Локально: `PUBLIC_DOMAIN=localhost`, порт наружу — в compose.
  - **Dev:** `docker-compose.override.yml` — при необходимости volumes + hot-reload для API и/или `next dev` вместо образа `web`.
  - **Healthcheck** в compose для `db`, `api`, `web`; `depends_on` с `condition: service_healthy` где уместно, чтобы nginx стартовал после готовности апстрима.

Доступ:

1. nginx: **`server_name`** совпадает с доменом из **`.env`** (`PUBLIC_DOMAIN` и т.п.); опционально жёсткая проверка заголовка `Host` только на этот домен; allowlist IP — по желанию.
2. Клиент передаёт ключ: заголовок `X-API-Key` или `Authorization: Bearer <key>` — **единый контракт** на фронте и бэке.
3. FastAPI middleware: нет ключа → `401`; невалидный ключ → `403` (или 401 — зафиксировать один раз в коде).
4. CORS и cookie (если появятся) — только для origin из **`PUBLIC_URL`** / `https://${PUBLIC_DOMAIN}`.

## 4. Архитектура (логическая)

```text
[Browser] → nginx (TLS, Host check)
              ├→ Next.js (статика/SSR) — в т.ч. вкладка «Уведомления» (прочитано/непрочитано)
              └→ FastAPI /api/*
                    ├→ PostgreSQL (вакансии, резюме, настройки, логи синков, анализы, уведомления)
                    ├→ hh.ru Open API (и позже другие источники)
                    ├→ LLM API (GigaChat / OpenAI) — ручной анализ + плановый батч (раз в час)
                    └→ Telegram Bot API — пуш при score ≥ порога
```

**Мульти-источник:** таблица `vacancy` с полем `source` (`hh`, …), уникальный ключ `(source, external_id)`.

**Синк hh.ru:** ежедневный прогон (по умолчанию **10:00**, таймзона из настроек/env) обходит **очередь активных сохранённых запросов**; в БД попадают **только новые** вакансии (**INSERT** при отсутствии пары `source` + `external_id`, без обновления дублей). Записи **старше 14 суток** (по **`published_at`** с hh или по согласованному полю «дата публикации») **удаляются из БД** и **не показываются** в ленте; связанные сущности (сопроводительные, анализы, уведомления) — **каскадно удалить** или пометить (предпочтительно CASCADE на FK).

**Markdown:** длинные текстовые поля (**резюме**, **сопроводительное**, **текст анализа LLM**, по желанию **описание вакансии** для UI) хранить и отдавать API в **Markdown**; на фронте — безопасный рендер (например `react-markdown` + ограничение тегов / sanitize). Сырой ответ hh с HTML при сохранении — конвертировать в MD или хранить отдельно `description_html` + `description_md` (зафиксировать один подход).

## 5. Функциональный бэклог

### 5.1 Инфраструктура и безопасность

- [ ] **Docker (полный стек):** `docker-compose.yml` поднимает **PostgreSQL + api + web + nginx**; `Dockerfile` для API и для `front/` (в образе фронта — **`pnpm install` / `npm ci` → `build` → `start`**); `.dockerignore`; в README — одна команда вроде `docker compose up --build -d`.
- [ ] **Docker (прод):** образы с тегами/версиями, не root в контейнерах приложений, лимиты ресурсов (`deploy.resources` / compose v3) при необходимости.
- [ ] **Alembic (must-have):** инициализация в репозитории бэкенда, ревизии с первой таблицей, команда миграций в entrypoint/Dockerfile API (`alembic upgrade head` перед стартом uvicorn), миграции в CI.
- [ ] Репозиторий ключей: переменная окружения `ADMIN_API_KEYS` (comma-separated) или таблица `api_keys` + хэш (argon2); для MVP допустим env.
- [ ] FastAPI: dependency `verify_api_key` на все защищённые роуты.
- [ ] CORS: только origin прод-домена (**тот же URL, что в `.env`** — `PUBLIC_URL` / домен).
- [ ] **nginx + `.env`:** переменная **`PUBLIC_DOMAIN`** (или эквивалент) в `.env.example`; шаблон конфига nginx + подстановка при старте контейнера (`envsubst`); в compose у сервиса `nginx` указать `env_file: .env` (или явный `environment`).
- [ ] nginx: прокси на `web` / `api`; при желании **basic** или mTLS — опционально поверх ключа.
- [ ] Секреты LLM и HH employer-токены (если появятся) — только env, не в БД в открытом виде.

### 5.2 Модель данных (черновик сущностей)

- [ ] `search_query` / `saved_search`: текст, регион (`area`), формат работы (`schedule`: remote, fullDay, …), `employment`, включено/выкл, `last_run_at`, `last_error`; участвует в **ежедневной очереди** синка (поле `interval` в UI — уточнить: только для матчинга/других джобов или убрать в пользу фиксированного 10:00).
- [ ] `vacancy`: сырой JSON + нормализованные поля (title, employer, salary, **`published_at`** — для **TTL 14 дней**, url, area, schedule, employment, skills…), `source`, `external_id`, `fetched_at`, `raw_payload` (JSONB); опционально **`description_md`** (и/или html) для отображения в Markdown.
- [ ] `resume`: название, поля и **`raw_text` / контент в Markdown**, активное по умолчанию, `created_at`.
- [ ] `cover_letter`: `vacancy_id`, `resume_id`, **текст в Markdown**, модель, `created_at`.
- [ ] `vacancy_analysis`: `vacancy_id`, `resume_id`, structured output (**числовой балл 0–100**, **категории** — список из фиксированной таксономии или enum, см. промпты/keyword taxonomy проекта), сильные стороны, пробелы, **блок «советы HR / IT lead»** (**Markdown**), `raw_ai_response`, `model`, `prompt_version`, `source` (`manual` | `scheduled`).
- [ ] `notification` / `match_alert` (или расширить `vacancy_analysis` флагами): связь с `vacancy_id` + `analysis_id`, **краткое summary для UI**, **`read_at` / `is_read`**, `telegram_message_id` (опционально), `created_at`. Одна запись = одно попадание в ленту уведомлений (в т.ч. после TG-пуша).
- [ ] `app_settings` (key-value) или одна строка настроек: **`hh_daily_sync_time`** (напр. `10:00`), **`hh_daily_sync_timezone`** (напр. `Europe/Moscow`), TTL **`vacancy_max_age_days`** = **14**, LLM provider, `llm_min_interval_seconds`, лимиты; **для фонового матчинга:** `match_analysis_interval_minutes` (напр. 60), пороги TG; **Telegram** `bot_token` + `chat_id` (только env или зашифровано — не в git), включение/выключение TG.

### 5.3 Синхронизация с hh.ru

- [ ] **Референс:** реализация HTTP к hh.ru — **строго по образцу `main.py`**: `https://api.hh.ru`, заголовок **`User-Agent`** (с контактом), те же query-параметры к `GET /vacancies`, обработка `items` / `pages` / `found`; перенос в сервис с заменой `requests` на **`httpx`** (или оставить единый стиль async).
- [ ] Для каждого **активного** `saved_search` в **очереди** (последовательно или с ограничением параллелизма): запросы к API, **в БД пишутся только новые** вакансии: **`INSERT … ON CONFLICT (source, external_id) DO NOTHING`** (или эквивалент); повторный синк **не дублирует** строки.
- [ ] **Ежедневный джоб сбора** (cron / APScheduler): по умолчанию **каждый день в 10:00** (время и TZ из **`app_settings`**), один прогон = обход всех включённых поисковых запросов в очереди. Ручной **«Синк сейчас»** из UI — опционально отдельным эндпоинтом.
- [ ] **Очистка «старше 14 дней»:** периодическая задача (тот же планировщик, можно сразу после дневного синка): удалить из `vacancy` записи, у которых **`published_at`** старше **14 суток** от текущей даты (или порог из `vacancy_max_age_days`); **GET /api/vacancies** и лента на фронте **не возвращают** удалённые строки. Каскад на `cover_letter`, `vacancy_analysis`, `notification` при удалении вакансии.
- [ ] Опционально `GET /vacancies/{id}` для полного описания при первом сохранении (тяжелее по лимитам — настраиваемо).
- [ ] Лог прогона: по каждому saved search — сколько **новых** вставлено, пропущено как дубликаты, ошибки, длительность.

*Примечание:* интервал «раз в N минут» в UI поисковых запросов может остаться для **других задач** или быть упрощён: **источник правды по hh-синку — ежедневный слот 10:00 + очередь запросов** (зафиксировать в настройках/UX, чтобы не плодить два конфликтующих расписания).

Отдельный промпт **scheduled match**: роль **HR lead / IT-рекрутер**, строгий **JSON-schema / Pydantic** с полями `score`, `categories[]`, `hr_advice`, `summary_for_notification`, и т.д.
- [ ] Retry при невалидном JSON + `json_repair` при необходимости.
- [ ] Очередь: если пользователь жмёт «Анализ» чаще лимита — ответ `429` + показать «доступно через X сек».

### 5.8 Плановый матчинг, Telegram, антиспам

- [ ] **Cron / APScheduler:** раз в N минут (по умолчанию **60**) — выборка вакансий для анализа (например новые за период или ещё без свежего `vacancy_analysis` для активного резюме); соблюдение **глобального лимита LLM** (очередь, задержка между запросами).
- [ ] После успешного анализа: если **score ≥ порога** из настроек — создать запись **`notification`**, отправить **Telegram** (текст: название вакансии, балл, 1–2 строки summary, ссылка на дашборд); **не дублировать** TG при повторном прогоне той же пары vacancy+resume (уникальный ключ или `last_notified_score`).
- [ ] Логирование ошибок TG (сеть, 403) без падения джоба.
- [ ] Настройки порога и интервала — **редактируемы в дашборде** (и дублируются/переопределяются env при необходимости).

## 6. Публичный API hh.ru — что реально использовать в аналитике

Без OAuth работодателя **число откликов обычно недоступно** (это данные работодателя). Для бесплатного анонимного доступа опирайтесь на поля из ответов:

- из списка/карточки: `name`, `employer`, `salary`, `area`, `published_at`, `alternate_url`, `schedule`, `employment`, `experience`, `snippet`, `key_skills` (в деталях), контакты часто скрыты;
- из `GET /vacancies/{id}`: полное описание, требования, опционально `branded_description`, счётчики **могут отсутствовать** для неавторизованного работодателя — **проверять фактическим ответом** и не обещать в UI метрики, которых нет.

Идеи аналитики без откликов: распределение по `area`, `experience`, `schedule`, топ работодателей, динамика по `published_at`, доля вакансий с зарплатой, вилки зарплат.
